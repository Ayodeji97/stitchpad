Measurements — visibility & the read-only view

Problem: tapping a saved measurement opens the always-editable form, so tailors fear accidental edits; measurements are also buried inside customer detail with no app-level entry point, and there is no way to share them. This page: ① three layout variants for a new read-only Measurement Detail screen, ② the three quick-access entry points you picked, ③ the share flow (image / PDF / WhatsApp text), each in light and dark.

"After saving the measurements… make it easier not to go directly to the editing Mode, it can just be saved as a note or table. Bcs if it go to edit mode u can accidentally edit it." — tester feedback
1 · Measurement detail (read-only) — light
Tap a saved measurement anywhere → this screen. Nothing on it is editable; Edit is an explicit action that opens the existing form. Delete moves here (overflow), out of the customer-detail row menu. JetBrains Mono finally gets used for the numbers.
V1 — Sectioned noteRecommended
Closest to the tester's "note or table". One card per body section, label left / mono value right. Scans top-to-bottom like the paper notebook tailors already keep. Sticky Edit + Share bar so actions never hide.

Post-save flow (fixes the tester's exact complaint):
Form → Save→ Read-only detail view + "Measurement saved" snackbar→ Back → customer detail
  • Behavior changes are additive: the edit form itself is untouched; the only changed behavior is that tapping a saved measurement (customer detail, order detail, post-save) lands on the new read-only view instead of the form. Create flow is unchanged.
  • Delete & rename move into the detail view's overflow (the row overflow on customer detail can keep them during a transition period, or drop them — to decide).
  • Sharing reuses proven infrastructure: OrderReceiptSharer's expect/actual render-to-image/PDF pattern, plus the existing WhatsAppLauncher for the text option.
  • Suggested slicing for the live app: PR 1 = read-only view + rerouted taps + post-save landing (the core fix) · PR 2 = sharing · PR 3 = entry points (dashboard shortcut, tape chips, search sub-results). Each independently shippable.
  • Locked (free-tier) customers: the read-only view is arguably fine to show locked customers too (read-only-visible principle) — Edit/Share would be gated. To decide.
